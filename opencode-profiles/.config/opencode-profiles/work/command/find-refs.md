КРИТИЧНОЕ ТРЕБОВАНИЕ (ОБЯЗАТЕЛЬНО):

* Используй MCP как ОСНОВНОЙ источник данных
* НЕ выполняй полный локальный обход файловой системы
* Локальное чтение файлов допустимо ТОЛЬКО:
  * после нахождения конкретного файла через MCP
  * и только для уточнения контекста
* Запрещено начинать анализ с Read, grep, или traversal по директориям


Инструкция:

1. Найди все использования $ARGUMENTS в кодовой базе:

   * импорты (`import`, `require`, dynamic import)
   * подключения через DI / registry / service locator
   * использование через прокси-слои (shared UI, adapters, wrappers)
   * для MFE: точки подключения (mount, bootstrap, loadRemoteModule, iframe и т.п.)

2. Используй MCP для построения графа зависимостей:

   * от $ARGUMENTS вверх по цепочке
   * до конечных точек входа:

     * страницы
     * роуты (react-router, next.js, custom router и т.д.)
     * server endpoints (если влияет на SSR)

3. Для каждого найденного использования:

   * раскрой всю цепочку:
     $ARGUMENTS → промежуточные компоненты/сервисы → страница/entrypoint
   * не останавливайся на shared-слоях — дойди до пользовательского сценария

4. Определи URL:

   * извлеки путь из роутера
   * если путь динамический — укажи шаблон (например `/vacancies/:id`)
   * если это MFE:

     * найди где он монтируется
     * определи URL хост-приложения

5. Обработай особые случаи:

   * ленивые загрузки (lazy, dynamic import)
   * feature flags / experiments
   * условные рендеры
   * модалки, виджеты без явного URL (привяжи к странице, где они открываются)

6. Сгруппируй результат по URL.

Формат ответа:

* URL: <url или шаблон>

  * Entry point: <страница / роут / контейнер>
  * Цепочка:
    $ARGUMENTS → … → конечный компонент
  * Файлы:

    * <path>: как используется $ARGUMENTS
  * Тип использования:

    * direct | indirect | mfe | lazy | conditional
  * Примечание:

    * (если есть особенности: флаг, эксперимент, неявный сценарий)

7. В конце добавь сводку:

   * список всех URL (чек-лист для тестирования)
   * места без явного URL (например: глобальные виджеты, shared layout)
   * потенциально пропущенные зоны (если анализ не полный)

Требования:

* не пропускать косвенные зависимости
* обязательно доходить до пользовательского сценария
* при неоднозначности — показывать несколько возможных цепочек
